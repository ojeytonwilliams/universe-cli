# Package Manager Lockfile Generation — Isolated Docker Runner

## Goal

Generate lockfiles inside an ephemeral Docker container with no bind mounts, so the host
filesystem receives only the files that belong in version control.

---

## Problem With the Current Approach

`docker-runner.ts` currently bind-mounts the entire project directory (`-v ${cwd}:/app`)
into the container. This means every file the package manager creates — the package store,
`node_modules`, caches — is written directly to the host. The caller cannot control what
comes back; it gets everything or nothing.

---

## Decisions

### 1. No bind mounts — use `docker cp` to move files

Files are transferred using `docker cp` rather than bind mounts or stdin/stdout pipes.
The container lifecycle is split across four explicit steps:

```
docker create   →  container ID
docker cp in    →  copy manifests into container
docker start    →  run the package manager command, wait for exit
docker cp out   →  copy lockfile out to host
docker rm       →  discard the container
```

The container is fully ephemeral. The package store, `node_modules`, and any other
side-effects vanish when it is removed.

A stdin/stdout tar pipe was considered but rejected: it requires binary stream handling
in Node.js (`spawn` + manual Buffer accumulation) and embeds archive logic in the
container command. `docker cp` delegates file transfer to Docker itself, keeps the
container command simple, and lets `execFile` (string I/O) be used throughout.

### 2. `manifests` is an explicit array, not implicit

Early drafts assumed `package.json` was always present and could be left out of the
config. This was rejected: as the tool expands to other languages (Rust, Go, Haskell,
etc.), the manifest file name is not constant. Every package manager entry must declare
its manifest files explicitly.

### 3. `lockfile` is an explicit field, not inferred

An alternative considered was listing all PM files in a single `pmFiles` array and
inferring the lockfile as "the entry that is not `package.json`". This was rejected
because inference is fragile — it breaks for any package manager where the manifest name
is not known in advance and makes the config harder to read. The lockfile is named
explicitly.

### 4. `devInstall` and `watchRebuild` are derived, not stored

Both fields are mechanical consequences of `manifests` and `lockfile`:

- `devInstall` → `COPY ${[...manifests, lockfile].join(' ')} ./\nRUN ${pm} install`
- `watchRebuild` → `[...manifests, lockfile].map(f => ({ path: './' + f }))`

Storing them separately creates three places to update when a lockfile name changes.
Generating them at composition time keeps the config as the single source of truth.

## Config Schema Change

`src/commands/create/layer-composition/layers/package-manager.json`

**Before:**

```json
{
  "pnpm": {
    "devCmd": ["pnpm", "run", "dev"],
    "devInstall": "COPY package.json pnpm-lock.yaml ./\nRUN pnpm install",
    "files": { ... },
    "pmInstall": "RUN corepack enable pnpm",
    "preinstall": "npx only-allow pnpm",
    "watchRebuild": [
      { "path": "./package.json" },
      { "path": "./pnpm-lock.yaml" }
    ]
  }
}
```

**After:**

```json
{
  "pnpm": {
    "manifests": ["package.json"],
    "lockfile": "pnpm-lock.yaml",
    "devCmd": ["pnpm", "run", "dev"],
    "files": { ... },
    "pmInstall": "RUN corepack enable pnpm",
    "preinstall": "npx only-allow pnpm"
  }
}
```

`devInstall` and `watchRebuild` are removed from the config and generated by the
composition service. `manifests` and `lockfile` together replace the implicit assumption
that `package.json` is always present.

### Generalises to other languages

```json
{
  "cargo": {
    "manifests": ["Cargo.toml"],
    "lockfile": "Cargo.lock",
    ...
  },
  "go": {
    "manifests": ["go.mod"],
    "lockfile": "go.sum",
    ...
  },
  "cabal": {
    "manifests": ["my-project.cabal", "cabal.project"],
    "lockfile": "cabal.project.freeze",
    ...
  }
}
```

## Docker Runner Changes

### New API

`docker-runner.ts` exposes two functions instead of the current `runCmd`:

```typescript
// Copies input files in, runs command, copies output files out
runCmdForFiles(
  cwd: string,
  cmd: string[],
  inputs: string[],   // relative filenames to copy from cwd into container
  outputs: string[],  // relative filenames to copy from container back to cwd
): Promise<void>

// Copies input files in, runs command, returns stdout
runCmdForStdout(
  cwd: string,
  cmd: string[],
  inputs: string[],   // relative filenames to copy from cwd into container
): Promise<string>
```

### Container lifecycle

```
const id = await docker create -w /app <image> sh -c "<pm-command>"

for each input:
  await docker cp <cwd>/<file> <id>:/app/<file>

await docker start -a <id>

for each output:
  await docker cp <id>:/app/<file> <cwd>/<file>

await docker rm <id>   // always, via finally
```

`execFile` (buffered string I/O) is sufficient throughout — no stream handling needed.

## Future Performance Improvements

The current design creates and discards a fresh container for every lockfile generation
run. This is correct and simple, but means the package manager re-downloads all
dependencies each time. Two improvements are deferred until there is evidence of a
performance problem:

### Reuse containers across runs

Instead of `docker create` + `docker rm` on every call, maintain a named container per
package manager. If the container already exists (check via `docker inspect`), skip
`docker create`. This avoids container creation overhead but does not help with
re-downloading packages (the container filesystem is reset on each `docker start`).

### Named volumes for the package store

Mount a named Docker volume at the package manager's cache/store path
(e.g. `pnpm-store:/root/.local/share/pnpm/store`). The volume persists independently of
the container lifecycle, so packages downloaded on the first run are available on every
subsequent run — even when the container is discarded and recreated. This is the most
impactful improvement and can be combined with or used independently of container reuse.

Both improvements are purely additive: they change `docker create` call-site arguments
and add an optional pre-flight check, without altering the overall `create`/`cp`/`start`/`cp`/`rm` structure.

The `lockfileName` field in `createPackageSpecifier` is sourced from `lockfile` in the
config, removing the duplication between the JSON config and the TypeScript classes.

The `PnpmRunner` and `BunRunner` interface signatures are unchanged — only the
implementations inside `defaultPnpmRunner` and `bunRunner` change.

---

## Invariants

1. **No bind mounts in the lockfile runner.** The host filesystem must not be writable
   by the container during lockfile generation.
2. **Explicit over implicit.** Every file the container needs as input, and every file
   the host expects as output, is declared in the config. Nothing is inferred from
   naming conventions.
3. **Config is the single source of truth for file names.** `devInstall`,
   `watchRebuild`, and `lockfileName` in the TypeScript classes are all derived from
   `manifests` and `lockfile` — they are never stored separately.
